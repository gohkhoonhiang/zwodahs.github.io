My Workflow

Background

I love sharing my workflow with people around me and I thought why not take it to a next level and just share it online. I believe that what I am going to share will be beneficial to anyone who wants to try out similar workflow.

I have been working on the terminal since 2012, started with vim as my editor and some basic cli-tools. I build my vimrc from scratch and have did a few rounds of refactoring on the way. Tmux was added to the workflow in late 2014 and it greatly improved my productivity.

I believe that as programmers, we should constantly be improving our tools and also seek out new tools to improve our workflow.

List of tools

My current list of tools that I use on a daily basis are

  1. Bash - My choice of shell.
  2. Tmux - Terminal Multiplexer. Highly customizable.
  3. Vim - Super Editor.

Bash

Bash is the choice of shell mainly because it was the one that I encountered first. When I first saw terminal, it was way back when I started programming and I couldn’t understand why anyone would switch from GUI to commandline. After working on the terminal, I can’t understand why anyone would move back.

I also stop using finder to navigate my filesystem. Using terminal allows me to get to somewhere faster or to find a specific file easier. If I need to drag the file somewhere, I can simply open it with -R option.

How ?

The easiest way to start using bash is to replace your daily workflow of finder with the basic commands like cd, ls, cat etc. This will get you used to the commandline without requiring you to do too much. After a while, you also should start thinking about making your bash looks nice and useful.

1. Start with a simple bash_profile

Start with a mindset that you hate typing. Create alias basic stuffs like ls -l to ll or aliasing it to even just l.

2. Learn all the shortcuts.

There are a few good shortcuts to learn. I am still trying to get some of them into muscle memory.

  1. Ctrl-f Ctrl-b : forward one char and backward one char. Why this when we can just use the arrow keys ? Arrow keys are kind of bad; It is less bad if you are using mac keyboard, but still not an excuse for you to use it
  2. Ctrl-p Ctrl-n : previous command and next command. Similar to just up arrow and down. See a pattern here?
  3. Ctrl-a Ctrl-e : go to start of line, go to end of line This is one that I haven’t got used to.
  4. Ctrl-r Ctrl-s : reverse/forward search for history
  5. Ctrl-w : delete word There are many more similar ones which I haven’t got into my muscle memory, like Ctrl-K for example.

While you are at it, change your capslock to ctrl. You don’t need the caplocks.

3. Learn bash scripting.

If you need to do something, try to do it in bash, using all the other commandline tools like cat, grep etc. Some automation in your terminal can really go a long way.

4. Share and learn

Share your dotfiles to the world (excluding all the sensitive information) and read other people dotfiles to learn new tricks. You can find mine here.

Tmux

Before I picked up tmux, I was already using vim for my C++ development. However, because I didn’t had tmux, I ended up having tons of terminal tabs on my iTerm.

What is Tmux

Tmux is a terminal multiplexer. It allows you to have tabs and splits in a single session. Why is this useful when you can just open multiple tabs/splits in iTerm/Terminal ?

Well, for one, if you are working on remote, it is really useful since you only need to ssh to the server once. On top of that, it allows you to copy text from one tab/window to another, without ever lifting your finger to use the mouse/trackpad. I really hate using trackpad.

The most important reason why tmux is useful, is that the splits and tabs are kept alive, even if you disconnect from the remote. I can’t emphasize how important this is, not just to ease the pain when you disconnect from the server due to the network failure, but also to allow you to work from different machines seamlessly.

How ?

So how to get started in tmux ? I would recommend the following steps.

1. Choose a binding.

The default key for tmux is Ctrl-B, but I use Ctrl-Q because it is easier to press. You don’t have to choose the same, just pick one, but preferbly one that you can reach easily, as you will be using it a lot.

2. Unbind all commands

This can be achieved by putting unbind-key -a in .tmux.conf in your home directory. Why should you unbind all commands ? This allow you to build the configuration slowly from scratch and only keep what you need. However, there are few of them that are important and need to be binded first.

  1. detach - allow you to exit tmux without closing all the tabs
  2. kill-window - kill the window that you are in.

That is all you need to ensure that you don’t get stuck in your tmux.

3. Bind commands that are useful to you.

To look at the full list of commands, you can use tmux list-commands. From there, you can see all the available commands in tmux. If you are unsure how to bind, look at my .tmux-conf and see how it is done.

The first few commands that might be useful are window splitting, new tabs and window resizing.

4. Share your .tmux.conf and learn from others.

When you are done configuring, it is a good idea to commit to a dotfile folder on github and let other people look and learn from it. You should also look at other conf file to see ideas that you can use.

Tips

One tip that might be useful when starting off is to have a binding that reload tmux conf, so you don’t have to restart tmux everytime you change something. It can be achieved by binding a key to source-file ~/.tmux.conf

Vim

What is Vim

Vim is a highly customizable editor that is available on the command line. If you are working on OSX or Linux, then it should already be available in your terminal.

Why Vim

Different people have different reasons why they switch to vim or choose vim in the first place. Some people grow up with it, and didn’t have to make the switch, others tried IDEs and various editors and realized that vim is the one. I belong to the latter.

So why do I continue to use vim even though many other “better” editors/IDE exists ? Here are my reasons.

No Distraction

Vim has a clean UI. When you run it, you only see your code any nothing else. Most of the other editor feels your screen with unncessary windows that leaves only a small part of the screen for your code.

Modal Editing

This is one of the things that once you get used to, you will never leave. In vim, you have 3 modes; Visual, Normal and Insert. In general,

  • Visual - Selecting stuffs
  • Normal - Moving around or applying commands
  • Insert - Adding texts.

The separation of modes allow you to think what you are trying to do and also allows for more specific key bindings for various modes, which brings me to next point

Key bindings

Normal IDEs requires tons of key bindings in the form of ctrl+alt+shift. Who use them ? seriously.. who ? Those are so HARD to use.

In Vim, you can use sequence of keystroke instead of combination. Pressing 2 keys one after another is definitely better than trying to press Ctrl-Shift ??.

On top of that, we have keybindings for everything. Visually select everything in “”, we have that - va" (inclusive of “) or vi"(exclusive of “), or if you just want to copy to clipboard, ya" and yi" will work.

Some vim users also like to change their caplocks to esc. This is because esc is the primary way to change the mode back to normal mode. However, I choose to bind my caplocks to ctrl instead. This is because I also use bash a lot and having Ctrl+? available is really handy. Instead of using esc, I am using Ctrl+[ to escape. It seems hard at first but I got used to it within 2 weeks.

No Mouse/Trackpad

Having to move to mouse or trackpad to do something sucks. When using vim, my hand is always on the keyboard. This means that there is little to no break of flow when I am programming.

Plugins

Having said all the reasons why I choose vim, the default vim isn’t enough. You will need to tweak it to make it your own and become one with it :D.

Sharing my configuration doesn’t make sense, because everyone have a different way of doing things. Instead, you can look at my vimfiles at github. They are heavily commented with the intention for people to learn from it. I will also showcase the list of plugins that I feel are the most important in the list I am using.

Fuzzy File Finder - CtrlP

There are many plugins for this, but my choice is CtrlP. It is the first that I encountered, and there is no need to replace it at the moment. Without this, opening a file involve manually typing the exact filename or using a file navigator. This is a MUST have.

Files/Folder Navigation

Nerdtree is my choice for this. Some people choose to use the builtin in vim but I think this provide a better view. I don’t use this as much now, after I started using CtrlP. I also started using Tmux + Bash to browse my files, so it might be time for NerdTree to go. Nevertheless, worth a mention.

Undo Tree

Ever undo so much that sometimes you wish you can go back to what you had undo ? Well, Gundo to the rescue. To be honest, you don’t need this plugin to find the edit that you want, you will just need to read the help file provided by vim. Gundo just provides a easy way to visualize your undo tree and find the edit you want.

Snippets

Snippets is something that I only started using recently. I can’t say much about it yet as I am still trying to get it into my muscle memory. I used to use vim-snippets but recently changed to UltiSnips. This main reason for the change is to allow it to work with my autocomplete engine.

AutoComplete

Vim default autocomplete was good enough for me for at least 2 years, before I found YouCompleteMe. It also works well with my snippets plugins, so that is a double win.

Motion

This is the last in the list, but will confuse anyone who don’t use vim and are new to vim. I will not go into details but easymotion will be really useful once you get used to vim’s basic motion keys.

Learning Vim

Many people have given their advices on how to learn vim. I will also give mine as well.

  1. Decide that you want to improve.
  2. Start small, if you haven’t try vim before use vimtutor.
  3. Don’t use the mouse, even if that makes you faster (in the short term).
  4. Don’t use the arrow key, ever (almost).
  5. Start with a minimum set of bindings, don’t download pre-bundled vim. There is too much there.
  6. Build your muscle memory. After a while, you stop thinking about how to do certain things and do it automatically.
  7. Steal vimrc. Learn from other people. Find new plugins.
  8. Keep learning and upgrading your vimrc.
  9. Share your vimfiles.
