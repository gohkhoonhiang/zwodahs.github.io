My Workflow

Background

I love sharing my workflow with people around me and I thought why not take it to a next level and just share it online. I hope that what I am sharing will be beneficial to others who wanted to try it out but don’t know how to start.

I have been working on the command line since 2012, started with vim as my editor and some basic cli-tools. Initially I was just working with python and Java and didn’t really mod my vim, until I started dealing with C++. I started building my vimrc from then as well as my dotfiles. Most of them are really simple stuffs until 2014 where I started working with web and remote servers. Tmux was also added to the workflow in late 2014 and it greatly improved my productivity.

As hackers/programmers/engineers, we should constantly be looking at ways to upgrade ourselves and our tools.

List of tools

My current list of tools that I use on a daily basis are

  1. Bash - My choice of shell.
  2. Tmux - Terminal Multiplexer. Highly customizable.
  3. Vim - Super Editor.

Bash

Bash is my choice of shell mainly because it was the first that I encountered and used. I tried zsh but I realized that I don’t really want to be installing it on every machine that I go to, so I decided to stick with bash.

When I first saw the command line, it was way back when I started programming. I couldn’t understand why anyone would use the command line when there is GUI. I couldn’t be more wrong; now I wonder why anyone would use GUI when command line is way faster.

My setup

There isn’t much set-up specifically for bash except for simple alias like g for git and l for ls -lGh. Yes, one letter alias. It is simple, the less you need to type the faster you get.

My bash_profile is a modular set up. It source various other files from the repo. This set up is important as I can turn on/off ‘features’ on different machines.

My default bash_profile is the one I used on most remote server. This is to allow minimum customization on new remote before I get started. I also have a nice setup.sh that set up everything once I clone my dotfiles. This include linking all my dotfiles, like .tmux.conf, .git(ignore|config), and also cloning my vimfiles and setting it up. So for every new machine I go to, I will just need to clone my dotfiles and run the setup script to set everything up.

How ?

The easiest way to start using bash is to replace your daily workflow of finder with the basic commands like cd, ls, cat etc. This will get you used to the command line without requiring you to do too much. Use open -R to reveal the folder in the finder if you need. After a while, you also should start thinking about making your bash looks nice and useful.

1. Start with a simple bash_profile

Start with a mindset that you hate typing. Create alias basic stuffs like ls -l to ll or aliasing it to even just l.

2. Learn all the shortcuts.

There are a few good shortcuts to learn. I am still trying to get some of them into muscle memory.

  1. Ctrl-f Ctrl-b : forward one char and backward one char. Why this when we can just use the arrow keys ? Arrow keys are kind of out of reach if you are using a normal keyboard. It is more natural for me to avoid arrow key since I am already doing that for vim.
  2. Alt-f Alt-b : forward/backward by word.
  3. Ctrl-p Ctrl-n : previous command and next command. This replaced the up and down arrow key. :D
  4. Ctrl-a Ctrl-e : go to start of line, go to end of line This is one that I haven’t got used to, but am actively trying to.
  5. Ctrl-r Ctrl-s : reverse/forward search for history
  6. Ctrl-w : delete word

While you are at it, change your capslock to ctrl, make your life easier on the terminal and a better person on the internet.

There are more shortcuts, but you should not try to memorize all of them. Instead, get them into your muscle memory one by one.

3. Learn bash scripting.

If you need to do something, try to do it in bash, using all the other command line tools like cat, grep etc. Some automation in your terminal can really go a long way.

For example, need to manually extract stuffs from a text file ? Use grep, cut, to print them out.

Bash scripting is not about learning the syntax of bash, but what are the tools available and how to combine them with pipes to do your task.

4. Share and learn

Share your dotfiles to the world (excluding all the sensitive information) and read other people dotfiles to learn new tricks. You can find mine here.

One of my favorite command I have in my dotfiles is h, which is a function that allow me to find previous commands using grep For example, h for curl will find me the for loop that I used previously to curl multiple places. It can be improved further but it works for now.

Tmux

Before I added tmux into my workflow, I was already using vim and command line for many of my workflow. I ended up having tons of tabs on my iTerm and it works. When I was introduced Tmux, it really changed how I do things and really improved my workflow by a lot.

What is Tmux

Tmux is a terminal multiplexer. It allows you to have tabs and splits in a single session. Why is this useful when you can just open multiple tabs/splits in iTerm/Terminal ?

If you are working on remote, it is really more convenient to have just a single session on the server. On top of that, it allows you to copy text from one tab/window to another, without ever lifting your finger to use the mouse/trackpad. I really hate using trackpad.

The most important reason why tmux is useful, is that the splits and tabs are kept alive, even if you are disconnected from the remote. I can’t emphasize how important this is. It reduce the pain when you get disconnected from the server and prevent you from losing work. In addtional to that, you can work on different machines and still get back the same tabs and windows.

How ?

So how to get started in tmux ? I would recommend the following steps.

1. Choose a binding.

The default key for tmux is Ctrl-B, but I use Ctrl-Q because it is easier to press. You don’t have to choose the same, just pick one, but preferably one that you can reach easily. You will be using it a lot.

2. Unbind all commands

This can be achieved by putting unbind-key -a at the start of .tmux.conf. You can look at my tmux.conf to see how I do it.

Why should you unbind all commands ? This allow you to build the configuration slowly from scratch and only keep what you need. However, there are few of them that are important and need to be binded first.

  1. detach - allow you to exit tmux without closing all the tabs
  2. kill-window - kill the window that you are in.

That is all you need to ensure that you don’t get stuck in your tmux. I binded detach to ‘d’ and kill-window to ‘z’. The logic for choosing ‘z’ to make it hard to reach so I don’t accidentally kill my window.

3. Bind commands that are useful to you.

To look at the full list of commands, you can use tmux list-commands. From there, you can see all the available commands in tmux. If you are unsure of how to bind the keys, you can look at my .tmux-conf and see how it is done.

The first few commands that might be useful are window splitting, new tabs, window resizing and navigation between the tabs and windows.

4. Share your .tmux.conf and learn from others.

When you are done configuring, it is a good idea to commit to a dotfile folder on github and let other people look and learn from it. You should also look at other conf file to see ideas that you can use.

Tips

One tip that might be useful when starting off is to have a binding that reload tmux conf, so you don’t have to restart tmux everytime you change something. It can be achieved by binding a key to source-file ~/.tmux.conf. As usual, look at my .tmux.conf to see what I mean.

Vim

I can talk about vim for hours. It is probably one of the best thing I found since programming.

I would recommend Emacs if modal editing is not your cup of tea, but do give modal editing a try. It really change how you think of editing, and really, the only drawback I see, is that I can no longer use other IDEs.

What is Vim

Vim is a highly customizable editor that is available on the command line. If you are working on OSX or Linux, then it should already be available in your terminal.

It is a really different beast compare to most of the other editors. For one, the modal editing will throw people off. When I first encountered vim, I wondered what this piece of shit is, and like most users, I don’t even know how to quit. It is frustrating and I can’t comprehend why anyone would use it. That was until I see what a power user can do with vim.

Why Vim

Different people have different reasons why they switch to vim or choose vim in the first place. Some people grow up with it, and didn’t have to make the switch, others tried IDEs and various editors and realized that vim is the one. I belong to the latter.

So why do I continue to use vim even though many other “better” editors/IDE exists ?

No Distraction

Vim has a clean UI. When you run it, you only see your code any nothing else. Most of the other editors fill your screen with unncessary windows that leaves only a small part of the screen for your code.

Modal Editing

This is one of the things that once you get used to, you will never leave. In vim, you have 3 modes; Visual, Normal and Insert. In general,

  • Visual - Selecting stuffs
  • Normal - Moving around or applying commands
  • Insert - Adding texts.

The separation of modes allow you to think what you are trying to do and also allows for more specific key bindings for various modes. It is so good that someone needs it on Emacs :D and created Evil.

Key bindings

Normal IDEs requires tons of key bindings in the form of ctrl+alt+shift. Who use them ? seriously.. who ? Those are so HARD to use.

In Vim, you can use sequence of keystroke instead of 3/4 keys combinations. Pressing 2 keys one after another is definitely better than trying to press Ctrl-Shift ??.

On top of that, we have keybindings for everything. For example va" and vi", which literally means visually select everything around/inside " If you want to jsut copy the text, ya" and yi"(yank) works as well. Of course, there are also da", di"(delete), ca"(change) .. okay you get the idea.

Some vim users also like to change their caplocks to esc, because esc is the primary way to change back to normal mode. However, I choose to bind my caplocks to ctrl instead. This is because I also use bash/tmux a lot and having Ctrl+? available is really handy. Instead of using esc, I am using Ctrl+[ to escape. It seems hard at first but I got used to it within 2 weeks.

No Mouse/Trackpad

Having to move to use the mouse or trackpad to do something really sucks. When using vim, my hand is always on the keyboard. This means that there is little to no break of flow when I am programming.

Plugins

Having said all the reasons why I choose vim, the default vim isn’t enough(sometimes). You will need to tweak it to make it your own and become one with it :D.

Sharing my configuration doesn’t make sense, because everyone have a different way of doing things. Instead, you should only look at my vimfiles after you are done learning the basic. They are heavily commented with the intention for people to learn/steal from it.

Instead, I will recommend some plugins that I think are worth mentioning and you should check them out after you started using vim on a daily basis. I chose this list because most people who come from IDE/GUI editors will always ask “but do you have this ?”.

Fuzzy File Finder - CtrlP

There are many plugins for this, but my choice is CtrlP. It is the first that I encountered, and there is no need to replace it at the moment. Without this, opening a file involve manually typing the exact filename or using a file navigator. This is a MUST have.

Files/Folder Navigation

Nerdtree is my choice for this. Some people choose to use the builtin in vim but I think this provide a better view. I don’t use this as much now, especially after I started using CtrlP. I also started using Tmux + Bash to browse my files, so it might be time to let NerdTree go. Nevertheless, worth a mention.

Undo Tree

Ever undo so much that sometimes you wish you can go back to what you had undo ? Well, Gundo to the rescue. To be honest, you don’t need this plugin to find the edit that you want, since vim already provides the functionality. You will just need to read the help file provided by vim. Gundo just provides a easy way to visualize your undo tree and find the edit you want.

This is the one plugin that I would ask all IDE users, “do you have this ?”

Snippets

Snippets is something that I only started using recently. I can’t say much about it yet as I am still trying to get it into my muscle memory. I used to use vim-snippets but recently changed to UltiSnips. This main reason for the change is to allow it to work with my autocomplete engine.

AutoComplete

Vim default autocomplete was good enough for me for at least 2 years, before I found YouCompleteMe. It also works well with my snippets plugins, so that is a double win.

Motion

This is the last in the list, but will confuse anyone who don’t use vim and are new to vim. I will not go into details but easymotion will be really useful once you get used to vim’s basic motion keys.

Learning Vim

Many people have given their advices on how to learn vim. I will also give mine as well.

1. Decide that you want to improve.

I can’t help you with this. You need to really believe in it and give it at least 2 weeks before deciding if it is useful for you.

2. Start small, really small

Don’t expect to be good with vim overnight. Start small, really small. If you haven’t try vim before, type vimtutor in your shell.

Start editing simple text with it. Don’t try to do actual work with it until you are really comfortable. The frustration of not being able to finish your work will make your quit. Try vim-adventures if you need to.

3. Don’t use the mouse

Don’t, just don’t, even if that makes you faster (in the short term).

4. Don’t use the arrow key, ever (almost).

Same as 3, just don’t.. well, except if you bind something to it :P. For me, I have rebind-ed my arrow keys to resizing my splits.

The important thing to remember is muscle memory. Once you build a certain muscle memory (like using the arrow keys or mouse), it will be really hard to change.

5. Start with a minimum set of bindings.

Don’t search online for vimrc or dotfiles for you to start with. The worse thing you can do is to download a pre-bundled vimrc, with tons of vim mapping and plugins that you (probably) will never use in the first 6 months. Start from scratch, and slowly build your muscle memory and vim bindings.

6. Build your muscle memory.

Did I mention this ? Yes build your muscle memory. At the start, you may need to think about how to do certain task, like moving around. But after a while, it will become natural and your mind will just think of the task and your fingers will automatically do it. Get it to a point where you don’t remember which key does what you want but you are still able to do it once you are on the keyboard. I always forget whether j is up or down when people ask me.

7. Steal vimrc.

This is Step 7. Do it after you done 1-6. Don’t skip ahead. Read Step 5 if you need to. But once you reach here, go ahead. Steal vimrc, not the whole of it, but part of it. Find out what people are doing and what plugins they are using. You may find interesting functions that they wrote but is not a plugin. If you do steal something, do leave a link in your vimrc so we can always find the source.

8. Keep learning and upgrading.

Always have one or two things that you are learning. Download vim cheatsheet and see what commands are available and use them. Once your muscle learn it, start learning something new.

Upgrade your vimrc. There is no reason why you should stop.

9. Share your vimfiles.

Share your vimfiles with everyone. Mine is here, and it is commented and organised for people to steal. Let me know if you have a nice plugin/config/function that I can steal.
